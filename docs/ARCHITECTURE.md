# ARCHITECTURE.md — معماریِ فنیِ «خوش‌حساب» (Khosh‑Hesab)

این سند **معماریِ فنیِ مرجع** اپ «خوش‌حساب» است: لایه‌بندی، جریانِ داده، انتخاب‌های فنی، قراردادهای پول/تاریخ، ساختار پوشهٔ کد، استراتژیِ تست و نقاطِ توسعه‌پذیری. هدف، یک پایهٔ مهندسیِ تمیز برای یک اپ موبایلِ **آفلاین‌فرست، خصوصی، رایگان و ساده** است که برای کافه‌بازار/مایکت منتشر می‌شود.

> **وضعیتِ تثبیت:** پروژه در **فاز ۰** است (داربست + تیم؛ هنوز کدِ اپ نوشته نشده). برخی تصمیم‌های زیربنایی هنوز با ADR نهایی نشده‌اند و در متن صریحاً با برچسبِ **«ADR-pending»** علامت خورده‌اند. این موارد **پیش‌فرضِ کاری**اند، نه قطعی؛ تثبیتِ نهایی‌شان در «دورِ ۱»ِ حلقه توسط `principal-architect` / `data-architect` انجام می‌شود (رجوع به [`CLAUDE.md`](../CLAUDE.md) و [`TEAM_AND_LOOP.md`](TEAM_AND_LOOP.md)).

---

## ۱) اصولِ راهنما (چرا این معماری)

سه شکایتِ مستندِ کاربران از رهبرِ فعلیِ بازار (اپ «دخل و خرج»، ۸۳هزار نصب، تنها **۳.۹★**) مستقیماً به سه الزامِ معماری ترجمه می‌شوند:

| شکایتِ کاربر (شواهدِ بازار) | الزامِ معماری |
|---|---|
| «از همان اول همه‌چیز پولی است» | هسته **همیشه‌رایگان و کامل**؛ هیچ مسیرِ حیاتی پشتِ paywall نیست. درآمد فقط لایهٔ نازکِ اختیاری روی هسته. |
| «برخلافِ اسمش ساده نیست / شلوغ است» | مسیرِ «ثبتِ تراکنش» باید در چند ثانیه و چند ضربه تمام شود؛ پیچیدگی در لایه‌های زیرین پنهان می‌ماند، نه در UI. |
| «نمودارها ابتدایی‌اند» | لایهٔ گزارش/نمودار یک **شهروندِ درجه‌یکِ معماری** است (`domain` مدلِ تحلیلی دارد، نه صرفاً نمایشِ خام). |

به‌علاوهٔ دو الزامِ زمینه‌ایِ ایران که شکلِ کلِ استک را تعیین می‌کنند:

1. **بدونِ هیچ وابستگی به Google Play Services (GMS).** GMS در ایران قابل‌اتکا نیست؛ پس FCM، Google Analytics، Crashlytics، ML Kit و هر SDK وابسته به GMS **ممنوع**اند. جایگزین‌ها: پوش = **پوشه**، آنالیتیکس = **متریکس** (حداقلی)، پرداختِ درون‌برنامه = **بازارپرداخت**، تبلیغِ اختیاری = **Rewardedِ تپسل**.
2. **آفلاین و خصوصی به‌صورتِ پیش‌فرض.** داده روی گوشی می‌ماند؛ **بدون لاگین، بدون سرورِ اجباری** برای هسته. این هم تمایزِ محصول است و هم سطحِ حملهٔ امنیتی/حریم‌خصوصی را تقریباً صفر می‌کند.

اصلِ کلانِ معماری: **سادگیِ بیرونی، انضباطِ درونی.** UI کم‌اصطکاک؛ زیرِ آن، یک لایه‌بندیِ تمیز و تست‌پذیر.

## ۲) نمای کلانِ لایه‌بندی

معماری یک **Clean Architecture**ِ یک‌سویه است: وابستگی‌ها فقط **رو‌به‌داخل** (به‌سمتِ `domain`) جاری می‌شوند. هیچ لایه‌ای از لایهٔ بیرونی‌ترِ خود چیزی نمی‌داند.

```
┌──────────────────────────────────────────────────────────────┐
│ presentation      ویجت‌ها، صفحه‌ها، ناوبری، تم RTL/شمسی        │  Flutter widgets
│   ▼ فقط از طریقِ provider/notifier با لایهٔ پایین حرف می‌زند    │
├──────────────────────────────────────────────────────────────┤
│ application       state، use-case orchestration، وضعیتِ صفحه   │  Riverpod (ADR-pending)
│   ▼ use-caseها را روی domain صدا می‌زند؛ UI-agnostic           │
├──────────────────────────────────────────────────────────────┤
│ domain            مدل‌های خالص، قواعدِ کسب‌وکار، اینترفیسِ repo  │  Pure Dart — بدونِ Flutter/SQL
│   ▲ هیچ وابستگیِ بیرونی ندارد؛ قلبِ پایدارِ سیستم               │
├──────────────────────────────────────────────────────────────┤
│ data              پیاده‌سازیِ repository، mapping مدل↔ردیف      │  Repository impls
│   ▼ اینترفیسِ domain را با منبعِ داده پر می‌کند                 │
├──────────────────────────────────────────────────────────────┤
│ db                drift/SQLite — جدول‌ها، کوئری، مهاجرت، DAO    │  drift (ADR-pending)
└──────────────────────────────────────────────────────────────┘
```

**قاعدهٔ آهنین:** ویجت‌ها SQL نمی‌زنند، و `domain` نه به Flutter وابسته است نه به drift. هر SQL فقط در لایهٔ `db`؛ هر دسترسی به داده از بیرون فقط از طریقِ **اینترفیسِ repository**ِ تعریف‌شده در `domain` و پیاده‌شده در `data`.

### نقشِ هر لایه

| لایه | مسئولیت | چه چیزی **نباید** اینجا باشد |
|---|---|---|
| **presentation** | ساختِ UI، چیدمان، تمِ RTL/شمسی، فرمتِ نمایش (تومان/ارقامِ فارسی)، ناوبری، واکنش به state | منطقِ کسب‌وکار، SQL، محاسبهٔ پول، I/O فایل |
| **application** | نگه‌داریِ state صفحه، صداکردنِ use-caseها، هماهنگیِ async، تبدیلِ خطا به وضعیتِ قابلِ‌نمایش | ویجت، رنگ/چیدمان، SQL خام |
| **domain** | مدل‌های immutable (`Transaction`, `Category`, `Account`, `Budget`)، قواعدِ خالص (جمعِ ریالی، طبقه‌بندی، بودجه)، **اینترفیسِ** `TransactionRepository` و … | هرچه به Flutter/drift/فایل بند باشد؛ این لایه با `dart test` خالص تست می‌شود |
| **data** | پیاده‌سازیِ repositoryها، نگاشتِ `entity row ↔ domain model`، ترکیبِ منابع (db + فایلِ پشتیبان) | قواعدِ کسب‌وکار (آن‌ها در domain‌اند)، ویجت |
| **db** | شِمای drift، DAOها، کوئریِ type-safe، مهاجرتِ نسخه‌بندی‌شده، رمزگذاریِ اختیاریِ فایل | منطقِ کسب‌وکار، فرمتِ نمایش |

## ۳) جریانِ داده — از ضربهٔ کاربر تا دیسک

مثالِ پایه: کاربر یک خرجِ «۸۵٬۰۰۰ تومان — خواروبار» ثبت می‌کند.

```
[۱] presentation                 کاربر مبلغ/دسته را وارد می‌کند → فراخوانیِ notifier
        │  ورودیِ متنی «۸۵٬۰۰۰ تومان» (ارقامِ فارسی)
        ▼
[۲] application (Notifier)        پارس و نرمال‌سازی → ریال صحیح (۸۵۰۰۰۰)
        │  ساختِ AddTransaction use-case input، set state = loading
        ▼
[۳] domain (UseCase + Model)      اعتبارسنجیِ قاعده (مبلغ>۰، دسته معتبر، تاریخِ معتبر)
        │  ساختِ Transaction (مبلغ: int ریال، تاریخ: DateTime میلادی، نوع: expense)
        ▼  فراخوانیِ TransactionRepository.add(tx)   ← اینترفیس
[۴] data (RepositoryImpl)         نگاشتِ domain model → drift companion row
        ▼
[۵] db (drift DAO)                INSERT type-safe → SQLite روی دیسکِ گوشی
        ▲
        └─ stream/Future برمی‌گردد؛ provider به‌روزرسانی می‌شود؛
           صفحهٔ «خلاصه/نمودار» که به همان stream گوش می‌دهد، خودکار rebuild می‌شود.
```

نکاتِ جریان:
- **پولِ ورودی در نخستین مرزِ ممکن (لایهٔ application) به `int` ریال تبدیل می‌شود** و از آن‌جا به بعد همه‌جا صحیح است. تومان/فرمت فقط در لبهٔ نمایش (presentation) دیده می‌شود.
- **خواندنِ گزارش** ترجیحاً **reactive** است: لایهٔ db کوئریِ drift را به‌صورتِ `Stream` می‌دهد؛ تغییرِ یک تراکنش، نمودار و خلاصه را بدونِ refreshِ دستی تازه می‌کند. این، حسِ «زنده و ساده» را به‌صورتِ معماری تضمین می‌کند.
- **خطاها** در لایهٔ application به وضعیتِ قابلِ‌نمایش تبدیل می‌شوند؛ هیچ exceptionِ خام به presentation درز نمی‌کند.

## ۴) انتخاب‌های فنی

| حوزه | انتخاب | وضعیتِ ADR | دلیلِ مهندسی |
|---|---|---|---|
| فریم‌ورک/زبان | **Flutter / Dart**، تک‌کدبیس | تثبیت‌شدهٔ پروژه | تجربهٔ یکدستِ RTL، عملکردِ بومی، اندازهٔ معقولِ apk بدونِ GMS، اکوسیستمِ بالغ |
| State management | **Riverpod** | **ADR-0002 pending** | تست‌پذیری بدونِ context، compile-time safety، مرزِ تمیز با application؛ تثبیت با `principal-architect` |
| پایگاه‌داده | **SQLite via `drift`** | **ADR-0003 pending** | کوئریِ type-safe، مهاجرتِ نسخه‌بندی‌شده، `Stream` reactive، codegen؛ تثبیت با `data-architect` |
| ناوبری | go_router (نامزد) | ADR-pending | مسیرِ اعلانی، deep-link آمادهٔ آینده |
| تاریخِ شمسی | `shamsi_date` | تثبیت‌شدهٔ پروژه | تبدیلِ میلادی↔شمسی برای نمایش |
| پوش | **پوشه** | تثبیت‌شدهٔ پروژه (بدونِ GMS) | جایگزینِ FCM در نبودِ GMS |
| آنالیتیکس | **متریکس** (حداقلی) | تثبیت‌شدهٔ پروژه | حریم‌خصوصی‌محور، بدونِ GMS |
| پرداختِ درون‌برنامه | **بازارپرداخت** | تثبیت‌شدهٔ پروژه | آنلاکِ یک‌بارهٔ نسخهٔ کامل |
| تبلیغِ اختیاری | **Rewardedِ تپسل** | تثبیت‌شدهٔ پروژه | ویدیوی پاداش‌محور، بدونِ بمباران |

> **چرا «pending» اما با پیش‌فرضِ روشن؟** برای جلوگیری از قفل‌شدنِ کار، استک یک پیش‌فرضِ کاریِ صریح دارد تا داربستِ کد جلو برود؛ اما هر سهٔ موردِ زیربنایی (state، db، ناوبری) پیش از قفلِ نهایی باید از حلقهٔ شور بگذرند و ADR بگیرند. اگر «دورِ ۱» جایگزینی پیشنهاد دهد (مثلاً sqflite به‌جای drift)، مرزِ `domain`/`data` چنان طراحی شده که تعویض، فقط لایهٔ `db`/`data` را لمس کند.

## ۵) آفلاین‌فرست و بدونِ‌GMS

- **منبعِ حقیقتِ داده روی خودِ گوشی است** (SQLite). هیچ فراخوانیِ شبکه‌ای در مسیرِ حیاتیِ ثبت/خواندن نیست؛ اپ بدونِ اینترنت کاملاً کار می‌کند.
- **بدونِ لاگین/حساب** برای هسته. هیچ هویتِ سرور-سمتی وجود ندارد ⇒ هیچ نشتِ PII، هیچ سطحِ حملهٔ احرازِ هویت.
- **هیچ importی از پکیج‌های وابسته به GMS مجاز نیست.** این یک قاعدهٔ قابلِ‌بازرسی است (می‌توان آن را در دروازهٔ کیفیت با یک تستِ ساده روی فهرستِ وابستگی‌ها enforce کرد).
- **لایه‌های اختیاریِ آنلاین** (پوشه/متریکس/بازارپرداخت/تپسل) همگی **پلاگین‌پذیر و قابلِ‌حذف**اند و پشتِ یک abstractionِ نازک قرار می‌گیرند؛ نبودشان نباید هسته را بشکند. این‌ها هرگز در `domain` ظاهر نمی‌شوند.
- **شبکه = فیچرِ لبه‌ای، نه ستونِ معماری.** هر قابلیتِ آنلاینِ آتی به‌صورتِ ماژولِ افزودنی طراحی می‌شود تا اصلِ آفلاین‌فرست نقض نشود.

## ۶) مدلِ پول — عددِ صحیحِ ریال (قاعدهٔ بی‌چون‌وچرا)

- **ذخیره و محاسبهٔ پول همیشه `int` به واحدِ ریال است. هرگز `double`/`float`.** نمایندگیِ پول با اعشارِ ممیزِ شناور به خطاهای گردکردنِ انباشتی منجر می‌شود؛ برای یک اپِ حساب‌وکتاب این غیرقابل‌قبول است.
- **تومان فقط واحدِ نمایش است.** تبدیلِ ریال→تومان (`÷۱۰`) صرفاً در لبهٔ presentation و در فرمترِ نمایش رخ می‌دهد. هیچ محاسبهٔ کسب‌وکاری روی تومان انجام نمی‌شود.
- **ارقامِ فارسی** فقط لایهٔ نمایش‌اند؛ ورودیِ کاربر در application به ارقامِ لاتین و سپس `int` ریال نرمال می‌شود.
- یک ابزارِ متمرکز (مثلاً `RialMoney` valueِ سبک یا توابعِ `MoneyFormatter`) مسئولِ تبدیل‌ها است تا منطقِ پول در یک‌جا متمرکز و تست‌شده بماند.

```dart
// domain — پول همیشه ریالِ صحیح
final tx = Transaction(amountRial: 850000, ...); // ۸۵٬۰۰۰ تومان
// presentation — تنها جایی که تومان/ارقامِ فارسی دیده می‌شود
Text(formatToman(tx.amountRial)); // «۸۵٬۰۰۰ تومان»
```

## ۷) مدلِ تاریخ و زمان — ذخیرهٔ میلادی، نمایشِ شمسی

- **ذخیره: میلادیِ ISO (`DateTime` / `YYYY-MM-DD`).** یک منبعِ حقیقتِ بدونِ‌ابهام برای ذخیره، مرتب‌سازی، بازه‌گیری و گزارش.
- **نمایش: شمسی** از طریقِ `shamsi_date`، تبدیل در لبهٔ presentation.
- **ورودیِ تاریخِ کاربر** (در صورتِ وجودِ date-picker شمسی) **سرور-سمت‌نداریم، اما همان اصل**: ورودیِ شمسی پیش از ذخیره در application به میلادی تبدیل می‌شود؛ هیچ‌گاه تاریخِ شمسیِ متنی در db ذخیره نمی‌شود.
- **زمان = محلیِ ایران.** برچسبِ زمانِ تراکنش با زمانِ محلیِ دستگاه ثبت می‌شود؛ از naive/UTCِ گمراه‌کننده در منطقِ نمایش پرهیز می‌شود.
- **RTL کامل** سراسرِ UI؛ تستِ golden مسئولِ صیانت از چیدمانِ راست‌به‌چپ است (بخش ۹).

## ۸) ساختارِ پیشنهادیِ پوشهٔ کد (`lib/`)

ساختار **feature-first در presentation/application**، اما **layer-first در domain/data/db** — تا هم پیمایشِ فیچر آسان باشد و هم مرزِ لایه‌ها سفت بماند. (این چیدمان ADR-pending است و در «دورِ ۱» نهایی می‌شود، اما با مرزهای بخش ۲ سازگار است.)

```
lib/
├─ main.dart                       ← bootstrap: تم، ProviderScope، ناوبری
├─ app/
│  ├─ app.dart                     ← MaterialApp، تمِ RTL، locale فارسی/شمسی
│  ├─ router.dart                  ← تعریفِ مسیرها (go_router — ADR-pending)
│  └─ theme/                       ← رنگ، تایپوگرافی، دیزاین‌توکن‌ها
│
├─ core/                           ← cross-cutting، بدونِ منطقِ فیچر
│  ├─ money/                       ← RialMoney، MoneyFormatter (ریال↔تومان، ارقامِ فارسی)
│  ├─ date/                        ← تبدیلِ شمسی↔میلادی، فرمترها
│  ├─ result/                      ← نوعِ Result/خطای دامنه
│  └─ utils/
│
├─ domain/                         ← خالص (Pure Dart) — بدونِ Flutter/drift
│  ├─ models/                      ← Transaction, Category, Account, Budget, Report…
│  ├─ repositories/                ← اینترفیس‌ها: TransactionRepository, BudgetRepository…
│  └─ usecases/                    ← AddTransaction, GetMonthlyReport, SetBudget…
│
├─ data/                           ← پیاده‌سازیِ repository + mapping
│  ├─ repositories/                ← TransactionRepositoryImpl…
│  ├─ mappers/                     ← row ↔ domain model
│  └─ backup/                      ← export/import فایلِ محلی (JSON، رمزگذاریِ اختیاری)
│
├─ db/                             ← drift (ADR-pending)
│  ├─ database.dart                ← تعریفِ DB، نسخه، مهاجرت
│  ├─ tables/                      ← transactions, categories, accounts, budgets
│  └─ daos/                        ← کوئریِ type-safe، Streamهای reactive
│
├─ features/                       ← presentation + application، گروه‌بندیِ فیچری
│  ├─ entry/                       ← «ثبتِ سریعِ تراکنش» (مسیرِ طلاییِ سادگی)
│  │  ├─ presentation/             ← صفحه‌ها/ویجت‌ها
│  │  └─ application/              ← notifier/provider، state
│  ├─ dashboard/                   ← خلاصهٔ یک‌نگاهی
│  ├─ reports/                     ← نمودارها و گزارش‌ها (فیچرِ تمایز)
│  ├─ budget/                      ← بودجه‌بندی
│  └─ settings/                    ← تم، پشتیبان‌گیری، آنلاک/درآمد
│
└─ integrations/                   ← لایه‌های اختیاریِ بدونِ‌GMS، همه پشتِ abstraction
   ├─ push_pushe/                  ← پوشه (اختیاری/قابل‌حذف)
   ├─ analytics_metrix/            ← متریکس (حداقلی)
   ├─ billing_bazaarpay/           ← بازارپرداخت (آنلاک)
   └─ ads_tapsell/                 ← Rewardedِ تپسل (اختیاری)
```

> نگاشتِ پوشه↔لایه: `domain/` و `data/` و `db/` همان لایه‌های بخش ۲‌اند؛ `features/*/presentation` لایهٔ presentation و `features/*/application` لایهٔ application است. `integrations/` همگی **بیرونی‌ترین** لایه‌اند و هرگز توسطِ `domain` import نمی‌شوند.

## ۹) استراتژیِ تست

تست، **دروازهٔ کیفیتِ پیش از هر انتشار** است؛ هیچ‌چیزِ قرمز منتشر نمی‌شود (رجوع به گامِ ۵ حلقه در [`TEAM_AND_LOOP.md`](TEAM_AND_LOOP.md)).

| سطح | چه چیزی را پوشش می‌دهد | چرا اینجا |
|---|---|---|
| **unit** | `domain` (قواعدِ پول، طبقه‌بندی، بودجه، تبدیلِ تاریخ)، `core/money`، `core/date` | بیشترین ارزش به ازای هزینه؛ `domain` خالص است و با `dart test` سریع و قطعی تست می‌شود |
| **widget** | ویجت‌های کلیدی، فرمِ ثبت، رفتارِ state در application | صحتِ تعامل و وضعیت بدونِ راه‌اندازیِ کلِ اپ |
| **integration** | جریانِ سرتاسری: ثبت→ذخیره در SQLite→بازتاب در گزارش؛ مهاجرتِ db؛ export/import پشتیبان | اطمینان از درست‌بودنِ مرزها و persist واقعی |
| **golden** | **RTL و نمودارها** — اسنپ‌شاتِ پیکسلی برای صیانت از چیدمانِ راست‌به‌چپ و رندرِ نمودار/ارقامِ فارسی | رگرسیونِ بصری در یک اپِ RTL/شمسی به‌سادگی رخ می‌دهد |

اصول:
- **پولِ ریالِ صحیح، تبدیلِ شمسی↔میلادی، و گردکردن** باید testهای واحدِ صریح داشته باشند (این‌ها منطقِ پُرریسک‌اند).
- **مهاجرتِ db** هر بار با تستِ مهاجرت از نسخهٔ قبلی پوشش داده می‌شود؛ هرگز فرضِ DB تازه نمی‌شود.
- **قاعدهٔ قابلِ‌enforce بدونِ‌GMS:** یک تستِ نگه‌بان که فهرستِ وابستگی‌ها را برای پکیج‌های وابسته به GMS بازرسی می‌کند (شکستن = قرمز).
- DoD یک فیچر: testهای سبز + بازبینیِ `qa-tester` + بدونِ رگرسیون + سندِ به‌روز + ثبت در `DECISION_LOG`.

## ۱۰) نقاطِ توسعه‌پذیری — افزودنِ گزارش/فیچرِ جدید

معماری برای **افزودن، نه بازنویسی** طراحی شده. الگوهای توسعهٔ متداول:

**الف) افزودنِ یک گزارش/نمودارِ جدید**
1. در `domain/models` مدلِ تحلیلیِ خروجی (مثلاً `CategoryBreakdown`) و در `domain/usecases` یک use-case (مثلاً `GetCategoryBreakdown`) تعریف کن.
2. اگر کوئریِ تازه لازم است، فقط در `db/daos` یک کوئریِ type-safe (ترجیحاً `Stream`) اضافه کن و در `data` آن را به مدلِ دامنه نگاشت کن.
3. در `features/reports` یک ویجت/notifier که به آن use-case گوش می‌دهد بساز. **هیچ تغییری در سایرِ فیچرها لازم نیست.**

**ب) افزودنِ یک نوعِ دادهٔ جدید (مثلاً «حسابِ بانکی» یا «بودجه»)**
- جدولِ drift + مهاجرتِ نسخه‌بندی‌شده در `db` → اینترفیسِ repository در `domain` → پیاده‌سازی در `data` → use-case → UI در `features/...`. مرزها تضمین می‌کنند هر تغییر در لایهٔ خودش می‌ماند.

**ج) افزودنِ یک لایهٔ اختیاریِ آنلاین/درآمد**
- زیرِ `integrations/` پشتِ یک abstractionِ نازک؛ هرگز در `domain`. حذف‌پذیری و آفلاین‌فرست حفظ می‌شود.

**نقاطِ توسعهٔ آینده (آگاهانه باز گذاشته‌شده):** بودجه‌بندی، طلب‌وبدهی، خروجیِ PDF/اشتراک، چنددستگی/همگام‌سازیِ اختیاری — همگی با همین الگو افزوده می‌شوند بدونِ لمسِ هسته. هرکدام که معماری‌محور باشد، یک ADR می‌گیرد.

## ۱۱) جمع‌بندی و موارد ADR-pending

این معماری سه وعدهٔ محصول را به‌صورتِ ساختاری پشتیبانی می‌کند: **رایگانِ واقعی** (هیچ مسیرِ حیاتی پشتِ paywall)، **سادهٔ واقعی** (مسیرِ ثبتِ کم‌اصطکاک + reactive UI)، **نمودارِ خوب** (گزارش به‌عنوانِ شهروندِ درجه‌یکِ `domain`) — به‌علاوهٔ آفلاین‌فرست و بدونِ‌GMS.

موارد **هنوز قطعی‌نشده (ADR-pending)** که باید پیش از قفلِ نهایی از حلقهٔ شور بگذرند:

| مورد | پیش‌فرضِ کاری | ADR | مالکِ تثبیت |
|---|---|---|---|
| State management | Riverpod | **ADR-0002** | `principal-architect` |
| پایگاه‌داده/ORM | drift روی SQLite | **ADR-0003** | `data-architect` |
| ناوبری | go_router | ADR آینده | `principal-architect` |
| چیدمانِ پوشهٔ نهایی | feature-first + layer-first (بخش ۸) | با ADRهای بالا | `principal-architect` |

> تا تثبیتِ این ADRها، کدِ داربست با همین پیش‌فرض‌ها جلو می‌رود؛ ولی مرزِ `domain`/`data`/`db` چنان نگه داشته می‌شود که تعویضِ هر انتخاب، تنها لایه‌های بیرونی را لمس کند، نه قلبِ دامنه.
